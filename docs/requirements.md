# SQL Processor 7주차 요구사항 정의서

## 1. 문서 목적

이 문서는 현재 루트 프로젝트를 기준으로 7주차 과제의 구현 범위와 완료 기준을 정의한다.

기준 프로젝트는 루트의 `SqlParser` 코드베이스이며, 6주차 SQL 처리기를 그대로 활용하되 7주차 목표인 `B+ 트리 인덱스`를 추가하는 방향으로 범위를 고정한다.

## 2. 요구사항 근거

- 근거 문서: `7주차 수요 코딩회 공지사항`
- 현재 기준 코드: 루트 `src/`, `include/`, `tests/`, `schema/`, `data/`
- 구현 언어: C
- 실행 환경: Docker 기반 Linux 개발 환경

## 3. 7주차 핵심 목표

이번 주 필수 목표는 아래와 같다.

- 기존 SQL 처리기의 `INSERT`, `SELECT`, `SELECT ... WHERE` 흐름을 유지한다.
- 각 레코드에 시스템이 `id`를 자동 부여한다.
- 부여된 `id`를 키로 사용하는 메모리 기반 `B+ 트리 인덱스`를 구현한다.
- `WHERE id = <value>` 조회 시 선형 탐색이 아니라 B+ 트리 인덱스를 사용한다.
- 대량 데이터 삽입 후 인덱스 조회와 비인덱스 조회의 성능 차이를 측정한다.
- 단위 테스트와 기능 테스트로 인덱스 동작을 검증한다.

## 4. 현재 기준 기능

현재 루트 프로젝트는 이미 아래 기능을 가진 상태를 기준으로 한다.

- SQL 파일 입력
- SQL 문장 직접 입력
- interactive REPL 입력
- `INSERT`
- `SELECT *`
- `SELECT column1, column2`
- `SELECT ... WHERE column = value`
- `schema/*.meta` + `data/*.csv` 기반 파일 저장

7주차 구현은 위 기능을 깨지 않고 확장해야 한다.

## 5. 포함 범위

### 5.1 SQL 처리 범위

- 기존 `INSERT` 유지
- 기존 `SELECT` 유지
- 기존 `SELECT ... WHERE column = value` 유지
- `WHERE id = <number>`일 때 B+ 트리 인덱스 사용
- `WHERE` 대상이 `id`가 아니면 기존 선형 탐색 유지

### 5.2 ID 관리 범위

- 각 테이블의 레코드에는 정수형 `id`가 자동 부여되어야 한다.
- `id`는 시스템 관리 컬럼으로 취급한다.
- 최종 저장되는 `id` 값은 시스템이 결정해야 한다.
- `id` 비교와 인덱스 키 비교는 정수 비교를 기준으로 해야 한다.
- 기존 SQL 처리기와의 연동을 해치지 않는 방향으로 `INSERT` 입력 형식을 유지하거나 확장할 수 있어야 한다.
- 새 레코드의 `id`는 기존 테이블의 최대 `id`보다 1 큰 값으로 부여한다.

### 5.3 인덱스 범위

- `id` 컬럼이 있는 모든 테이블에 대해 테이블별로 하나의 `id` 인덱스를 자동으로 유지한다.
- 인덱스 자료구조는 메모리 기반 B+ 트리다.
- 인덱스 키는 `id`다.
- 인덱스 값은 해당 레코드를 빠르게 찾기 위한 위치 정보다.

### 5.4 성능 테스트 범위

- 최소 1,000,000건 이상의 레코드를 삽입할 수 있어야 한다.
- 대량 레코드를 반복 가능하게 생성할 수 있는 입력 생성 경로 또는 데이터 생성 도구가 있어야 한다.
- 같은 입력 파라미터 또는 같은 시드를 사용하면 같은 데이터셋을 다시 생성할 수 있어야 한다.
- `WHERE id = <value>` 조회 시간을 측정한다.
- `WHERE other_column = <value>` 조회 시간을 측정한다.
- 두 결과를 같은 실행 환경에서 비교 가능해야 한다.
- 성능 측정은 별도 실행 경로 또는 별도 벤치마크 도구로 제공해도 된다.

### 5.5 테스트 범위

- B+ 트리 단위 테스트
- ID 자동 부여 테스트
- `INSERT` 후 인덱스 등록 테스트
- `WHERE id = <value>` 인덱스 조회 테스트
- 일반 `WHERE` 선형 탐색 테스트
- 대량 삽입 후 기본 기능 테스트

## 6. 제외 범위

아래 기능은 7주차 필수 범위에 포함하지 않는다.

- 디스크 기반 B+ 트리
- 복합 인덱스
- `UPDATE`
- `DELETE`
- `JOIN`
- `ORDER BY`
- `GROUP BY`
- 복합 `WHERE`
- 범위 검색 최적화
- 동시성 제어
- 트랜잭션
- 다중 테이블 인덱스 조인
- 네트워크 서버화

## 7. 전제 조건

- 테이블 스키마는 `schema/<table>.meta`로 존재한다고 가정한다.
- 실제 레코드 파일은 `data/<table>.csv`로 존재한다고 가정한다.
- 각 테이블의 첫 줄은 컬럼 헤더다.
- `id` 컬럼을 포함한 모든 테이블은 7주차 자동 인덱싱 대상이다.
- 기존 6주차 테이블 구조를 그대로 사용할 수 없는 경우, 7주차 대상 테이블 스키마와 예제 데이터는 `id` 컬럼을 포함하도록 정비해야 한다.
- 현재 프로젝트는 append 중심 흐름이며 `UPDATE`, `DELETE`가 없으므로 레코드 위치 정보는 삽입 후 안정적으로 유지된다고 가정한다.

## 8. 기능 요구사항

### 8.1 INSERT 요구사항

- 사용자는 7주차 대상 테이블에 대해 기존 SQL 처리기와 호환되는 `INSERT`를 실행할 수 있어야 한다.
- 실행기는 새 `id`를 자동 생성해야 한다.
- 저장 전 새 레코드를 CSV 행으로 구성해야 한다.
- 저장 직후 해당 `id`를 B+ 트리에 등록해야 한다.
- 저장 성공 시 인덱스와 CSV가 서로 일치해야 한다.
- CSV 저장 후 인덱스 등록에 실패하면 현재 실행은 오류를 반환해야 하며, 이후 인덱스는 CSV 기준 재구성을 통해 일관성을 회복할 수 있어야 한다.

예시:

```sql
INSERT INTO student (department, student_number, name, age)
VALUES ('컴퓨터공학과', '2024001', '김민수', 20);
```

### 8.2 SELECT 요구사항

- 기존 `SELECT *` 동작을 유지한다.
- 기존 특정 컬럼 조회 동작을 유지한다.
- `WHERE`가 없으면 기존처럼 전체 스캔으로 출력한다.
- `WHERE id = <number>`이면 인덱스를 사용한다.
- `WHERE id = <value>`에서 `<value>`는 정수여야 하며, 정수가 아니면 오류 처리해야 한다.
- `WHERE name = 'value'` 같은 일반 조건은 기존 선형 탐색을 유지한다.

예시:

```sql
SELECT * FROM student WHERE id = 1000;
SELECT name, age FROM student WHERE department = '컴퓨터공학과';
```

### 8.3 인덱스 재구성 요구사항

- 프로그램 시작 후 특정 테이블의 인덱스가 아직 메모리에 없으면 CSV를 읽어 인덱스를 재구성할 수 있어야 한다.
- 이미 메모리에 있는 인덱스는 같은 프로세스 안에서 재사용할 수 있어야 한다.
- REPL 또는 벤치마크 실행 중 반복 조회에서는 같은 인덱스를 재사용해야 한다.
- 인덱스 재구성 중 중복 `id`, 정수가 아닌 `id`, 누락된 `id`를 만나면 오류로 처리해야 한다.

## 9. 데이터 저장 규칙

- 기존 저장 포맷은 CSV를 유지한다.
- `id`는 CSV 헤더의 실제 컬럼으로 존재해야 한다.
- 인덱스는 메모리 구조이므로 CSV 파일이 영속 저장의 기준이다.
- 프로그램이 시작될 때마다 인덱스는 CSV를 기준으로 다시 만들 수 있어야 한다.
- CSV와 메모리 인덱스가 불일치할 수 있는 상황에서는 CSV를 기준으로 인덱스를 다시 구성해야 한다.

## 10. 오류 처리 요구사항

- `id` 컬럼이 없는 테이블은 인덱스 대상 테이블로 사용할 수 없어야 한다.
- 존재하지 않는 테이블 조회 시 오류 처리해야 한다.
- 존재하지 않는 컬럼 조회 시 오류 처리해야 한다.
- 잘못된 `WHERE` 문법은 오류 처리해야 한다.
- `WHERE id = <value>`에서 정수가 아닌 값은 오류 처리해야 한다.
- 인덱스 조회 시 해당 `id`가 없으면 빈 결과를 반환해야 한다.
- CSV 내용과 인덱스 재구성 중 읽은 컬럼 수가 맞지 않으면 오류 처리해야 한다.
- 인덱스 재구성 중 중복 `id`, 정수가 아닌 `id`, 누락된 `id`를 만나면 오류 처리해야 한다.

## 11. 테스트 요구사항

### 11.1 단위 테스트

- B+ 트리 노드 삽입
- B+ 트리 탐색
- 노드 분할
- 중복 키 처리 정책
- `id` 자동 생성
- CSV 오프셋 계산
- `WHERE id = value` 분기 결정
- 인덱스 재구성 중 잘못된 `id` 검증

### 11.2 기능 테스트

- `INSERT` 후 CSV 저장과 인덱스 등록이 모두 성공한다.
- `SELECT ... WHERE id = value`가 정확한 한 행을 반환한다.
- 일반 `WHERE`는 기존 방식대로 동작한다.
- 프로그램 재시작 후 인덱스를 재구성해 같은 `id` 조회 결과를 얻는다.
- CSV 저장 후 인덱스 등록 실패가 발생하면 다음 인덱스 접근 시 CSV 기준 재구성으로 복구할 수 있다.
- 대량 삽입 후 인덱스 조회와 선형 조회의 성능 비교가 가능하다.
- 같은 입력 파라미터로 벤치마크를 다시 실행하면 같은 데이터셋을 재생성할 수 있다.

## 12. 완료 기준

아래 조건을 모두 만족하면 7주차 필수 구현 완료로 본다.

- 기존 SQL 처리기의 핵심 흐름이 유지된다.
- `INSERT` 시 `id`가 자동 생성된다.
- 생성된 `id`가 B+ 트리에 등록된다.
- `WHERE id = <value>` 조회가 인덱스를 사용한다.
- 비인덱스 컬럼 조회는 기존 선형 탐색으로 유지된다.
- 1,000,000건 이상 삽입 가능한 성능 테스트 경로가 존재한다.
- 단위 테스트와 기능 테스트가 존재한다.
